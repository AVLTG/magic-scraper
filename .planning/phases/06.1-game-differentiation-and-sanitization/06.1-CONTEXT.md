# Phase 06.1: Game differentiation and sanitization - Context

**Gathered:** 2026-04-11
**Status:** Ready for planning

<domain>
## Phase Boundary

Harden the game-tracking surface landed in Phase 6 along three axes:

1. **Differentiate ~20 spreadsheet-imported legacy games** from future user-entered games so Phase 7 stats can exclude them from combo-rate and deck-based queries. The imported rows already carry a free-text "Ported from Spreadsheet" note; 6.1 formalizes this into a structural flag and backfills the production data.
2. **Sanitize game-entry input** by preventing the same player from being listed twice in a single game — both by filtering already-used names out of the Combobox dropdowns and by replacing the "Add as new player" affordance with a disabled collision-notice row when the user types a conflicting name.
3. **Add a filter toolbar to the `/games` history list** so users can narrow by winner, player count (2/3/4), and participating players, with AND-across-types / OR-within-multi-select semantics and ephemeral client-side state.

Phase 6.1 does not touch Phase 7 (stats dashboard) or Phase 8 (admin tools), but the `isImported` flag it introduces is a direct input into Phase 7's stat queries. The filter UI is additive to the Phase 6 `/games` list and reuses its existing client-side "load all games, filter in memory" pattern (06 D-12).

Out of phase: any change to existing Phase 6 CRUD behavior, new stats, charts, visible imported badges, pagination, server-side filters, Phase 7 filter-integration.

</domain>

<decisions>
## Implementation Decisions

### Import Marker (Schema + Backfill)

- **D-01:** **Add `isImported Boolean @default(false)` to the `Game` Prisma model.** Binary distinction, direct stat query (`WHERE isImported = false`), no enum ceremony. Additive schema change only — matches Phase 5 D-16 ("additive only, no ALTER, no DROP"). All newly-created games via `POST /api/games` get `isImported = false` without any form change (Prisma default).
- **D-02:** **The flag is hidden from every UI surface.** Never rendered as a badge, column, checkbox, or form field. The games list and edit form look identical to the current Phase 6 UI. The only human-visible indicator of an imported game is the existing `"Ported from Spreadsheet"` text in the `notes` field, which is left untouched. Rationale: editability is not restricted to admins in the current codebase (regular shared-password users can edit any game), and making the flag editable would let users accidentally flip stat inclusion for live games.
- **D-03:** **Backfill via one-off SQL against production Turso.** Plan step runs `turso db shell` with `UPDATE games SET isImported = 1 WHERE notes LIKE '%Ported from Spreadsheet%'`. Matches Phase 5 D-14 manual-migration pattern for Turso. After the update, a `SELECT COUNT(*) FROM games WHERE isImported = 1` verification step confirms ~20 rows were flagged. If the SQL path is blocked for any reason, the fallback is manual edit through the UI — but this requires temporarily exposing the flag in the form, so SQL is strongly preferred.
- **D-04:** **Notes field is not cleaned up.** The `"Ported from Spreadsheet"` substring stays in the `notes` column even after the column-based flag lands. Two redundant indicators for one fact is fine; zero risk of losing user-written context; keeps the human-readable marker visible in the list. The Notes column in the games table continues to display it as-is.
- **D-05:** **Apply the same `isImported Boolean @default(false)` to local dev via `prisma db push`**, matching Phase 5 D-13. Production gets the column via `turso db shell` ALTER TABLE (same path as Phase 5 prod migration) followed by the backfill UPDATE from D-03.

### Stats Exclusion Semantics (Input to Phase 7)

- **D-06:** **Imported games are excluded from combo-rate and deck-based stats only.** Phase 7 stat queries for `wonByCombo` percentages and any per-deck metric (deck win rate, deck participation pie, games-per-deck) must include `WHERE isImported = false` (or equivalent Prisma filter). These fields are treated as missing/unreliable for imported rows even if the column is non-null.
- **D-07:** **Imported games ARE counted in all other stats.** Player win rate, screwed rate, weekly game frequency, and "most likely to play" participation all include imported rows. Rationale: the `date`, `playerName`, `isWinner`, and `isScrewed` fields were accurate in the source spreadsheet — only combo and deck info were missing.
- **D-08:** **Imported games remain fully visible in the `/games` list.** No hide-by-default, no show-imported toggle, no "imported-only / live-only" filter chip. They render identically to live games. The filter toolbar (see Filter UI section) operates uniformly across all games regardless of import source.
- **D-09:** **The `isImported` flag does not gate the new filter toolbar's behavior.** Winner / player count / participating-players filters operate on all games without consulting `isImported`. The flag is exclusively a Phase 7 stats concern plus a row-creation default.

### Duplicate-Player Prevention (GameForm UX)

- **D-10:** **Already-filled player names are filtered out of OTHER rows' Combobox dropdowns.** When Player 1 = "Amirali", the dropdowns presented to Player 2/3/4 omit "Amirali" from their `items` list. Clearing or changing Player 1 restores "Amirali" to the other dropdowns. This is a pure derived computation from the current `state.rows` — no new state, no API changes.
- **D-11:** **Each row's own dropdown always includes its current value.** So editing an already-filled row doesn't hide the player's current name from themselves. Filter logic: `items.filter(i => !otherFilledRows.includes(i))` where "other" means all rows except the one currently showing the dropdown.
- **D-12:** **When a user manually types a name that collides with an already-filled row, the Combobox's "Add 'xyz' as new player" affordance (Phase 6 D-08) is REPLACED by a disabled, non-clickable dropdown row reading `"Player already in game"`.** This is the complete warning channel — no inline text under the field, no top-of-form banner, no toast. The dropdown itself is the feedback. Keyboard navigation (↑↓) skips the disabled row.
- **D-13:** **The check is live on every keystroke / state change.** Consistent with Phase 6 D-09 (seed-once, client-side filter) — no network calls, just a derived computation. The moment Player 1 is set, Player 2's dropdown updates; the moment Player 2's typed value collides with Player 1's name, the dropdown swaps in the disabled row.
- **D-14:** **Server-side enforcement via zod refine on `gameSchema.participants`.** Even though the form prevents dupes client-side, the API route should also reject duplicate player names within a single game's `participants` array — defense-in-depth for Phase 6 D-29 ("if it passed zod, it's safe to insert"). A `.refine(arr => new Set(arr.map(p => p.playerName.toLowerCase())).size === arr.length, "duplicate player names not allowed")` on the participants array. Case-insensitive comparison so "Amirali" and "amirali" collide.
- **D-15:** **The Combobox component gains an `excludeItems?: string[]` prop** (or similar) so the GameForm can pass per-row exclusion lists without duplicating filter logic. The existing `filterItems` pure helper is extended or wrapped with an exclude step. The "Add as new player" suppression + disabled-row swap is a new branch in the Combobox render that reacts to the exclusion list.

### Filter Toolbar (Games History List)

- **D-16:** **Inline toolbar renders above the games table.** Lives in `src/app/games/page.tsx`, rendered between the "Games / Log game" header row and the `<table>`. Horizontal flex layout with three controls: `[Winner dropdown] [Player count dropdown] [Players multi-select]`. Matches the existing Phase 6 compact layout; no collapsible panel, no sidebar.
- **D-17:** **Filter combine logic: AND across filter types, OR within multi-select.** Example: "Winner = Amirali" AND "PlayerCount = 4" AND "(Alice played OR Bob played)". The standard faceted-filter behavior users expect. Within the Players multi-select, a game passes if ANY of the selected players participated (not ALL).
- **D-18:** **Player count filter options: 2, 3, and 4.** Even though typical play is 2 or 4, 3-player games are a valid GameParticipant configuration per GAME-01 (1-4 players) and must be filterable if they exist. "1 player" is omitted — a solo Magic game is a data-entry mistake and doesn't deserve a filter bucket.
- **D-19:** **Winner filter source: distinct `playerName` values from `participants.filter(p => p.isWinner === true)` across the currently-loaded games list.** Dynamic — no dead options. Alphabetized. If zero games exist (empty state), the Winner filter is rendered disabled / greyed out.
- **D-20:** **"Players participated" multi-select source: distinct `playerName` values from ALL participants across the currently-loaded games list.** Also dynamic and alphabetized. Includes both winners and non-winners. Uses the existing Combobox pattern or a simple checkbox popover — planner's discretion.
- **D-21:** **Filter state is ephemeral client-side `useState`** in the `GamesPage` component. No URL querystring, no `useSearchParams`. Filters reset on page reload or navigation. Consistent with Phase 6's no-URL-state convention (nothing in the codebase currently uses URL params for UI state).
- **D-22:** **Filters are applied via a derived value**, not via refetch. `const filteredGames = useMemo(() => games.filter(matchesAllFilters), [games, winnerFilter, countFilter, playerFilters])`. Zero network calls, zero loading states. The existing `GET /api/games` endpoint is unchanged — Phase 6 D-12 ("load all games, filter client-side") is the foundation.
- **D-23:** **Filter controls have a visible "Clear filters" affordance** when any filter is active (e.g., a small "Clear" button or an X on each filter pill). Prevents users getting stuck in a filtered view with no obvious reset path.

### Claude's Discretion

- Exact visual styling of the filter toolbar (Tailwind classes, spacing, colors) — follow the existing `/games` page Tailwind patterns.
- Whether the Players multi-select uses a reused Combobox, a checkbox dropdown, or a pill-style filter with removable chips — planner picks the simplest option that matches the app's visual language.
- Whether the "Clear filters" affordance is a button next to the toolbar, X buttons on individual filter pills, or both.
- Exact wording of the disabled "Player already in game" row (could also be "Already in this game" or similar — the user's example wording is "Player named 'X' is already listed in this recording"; planner picks something that reads naturally in the dropdown context).
- Whether the zod duplicate-player refine message surfaces as a form-level error banner or is swallowed because the client already prevents submission — planner can decide based on the final UX test.
- Whether the empty-state "No games match your filters" message is distinct from the existing "No games logged yet" empty state (recommended: yes — they mean different things).
- The exact SQL verification step after backfill (just count, or count + spot-check a sample row) — planner's discretion.
- Whether to write a unit test for the `excludeItems` Combobox prop or rely on integration testing through the GameForm — planner's discretion given existing Phase 6 test conventions.

## Existing Code Insights

### Reusable Assets
- **`src/app/components/combobox.tsx`** — Already has `filterItems(items, inputValue)` and `shouldShowAddNew(items, inputValue)` pure helpers. 6.1 extends these with an `excludeItems` parameter (or wraps them) so the "Add new" affordance can be suppressed and replaced with a disabled row.
- **`src/lib/validators.ts` gameSchema** — Already validates 1-4 participants, trims player names, enforces length caps. 6.1 adds a `.refine()` for duplicate detection; no structural change, just one additional rule.
- **`src/app/games/page.tsx`** — Already loads all games client-side and renders them in a table. 6.1 adds filter state + a `useMemo` derived filtered list + toolbar JSX; the table rendering loop is untouched except it iterates over `filteredGames` instead of `games`.
- **`src/app/games/game-form.tsx`** — Already manages per-row state and passes `items` to each Combobox. 6.1 adds a derived `excludeItemsForRow(rowIndex, state)` helper that returns all other rows' non-empty player names, then passes it to the matching Combobox instance.
- **Phase 5 Turso migration pattern** — `turso db shell < migration.sql` is an established production-migration path. 6.1's ALTER TABLE + backfill UPDATE fits cleanly.

### Established Patterns
- **Client-side-only list UX** — No URL params, no server-side filters, no refetch on state change. Filter toolbar follows the same convention.
- **Derived state via useMemo** — Not currently used aggressively, but a natural fit for the filter layer. No new libraries needed.
- **Zod refine for cross-field validation** — Not yet used in `gameSchema`, but zod's `.refine()` / `.superRefine()` is the idiomatic place for cross-participant rules.
- **Prisma additive migrations** — Phase 5 established the pattern of adding columns without data backfill in the migration itself; data backfill is a separate explicit step. 6.1 follows this.
- **Phase 6 D-29 "sanitization lives in zod"** — The server-side dup-player check MUST be in `gameSchema`, not in the route handler. Route just calls `.parse()` and trusts the result.

### Integration Points
- **Schema migration** — `prisma/schema.prisma` gains one line on `Game`: `isImported Boolean @default(false)`. `prisma db push` handles local dev. Production gets a `turso db shell` ALTER + UPDATE as a plan step.
- **Combobox prop extension** — `ComboboxProps` gains `excludeItems?: string[]`. Backward compatible (optional, defaults to empty). All existing call sites continue to work.
- **GameForm row-to-exclude mapping** — Per-row derived function. When rendering the Combobox for row `i`, compute `excludeItems = state.rows.filter((_, idx) => idx !== i).map(r => r.playerName.trim()).filter(Boolean)` and pass it in.
- **No new API endpoints** — Zero route file creation. 6.1 modifies existing `gameSchema` and existing `/games` page; all other touch points stay stable.
- **Filter toolbar component** — Either inlined in `page.tsx` or extracted to `src/app/games/filter-toolbar.tsx` (planner's call). No new dependencies either way.

<specifics>
## Specific Ideas

- **Disabled dropdown row for dup-player collision** — User's exact phrasing: "It could show up in the dropdown in replacement of the 'Add Amirali as new player'. Just an unclickable dropdown item that says 'Player already in game'". This is the complete warning channel — no additional inline text, no top banner, no toast. Implement the disabled row with visual dimming (muted color) and `aria-disabled="true"`; keyboard ↑↓ must skip it.
- **"Ported from Spreadsheet" note stays visible** — Even after the column lands, the notes text is the user's deliberate signal. Do not strip, do not replace with a badge, do not hide the notes column for imported rows. User's exact phrasing: "ported from spreadsheet note should be the only thing people see".
- **Imported data is treated as missing `wonByCombo` + `deckName`** — User's exact clarification: "All imported data is counted as missing 'Won by combo' and 'Deck Name' fields, even if they are checked or have a name listed." Stored values may be non-null (some imported rows may have deck names), but Phase 7 stat queries MUST ignore them. This is a semantic exclusion, not a DB constraint.
- **Manual-UI backfill is an acceptable fallback** — User's note: "I am fine with the one-off sql in a plan step but if needed I dont mind doing the manual UI edit either. 20 is not that crazy." This means if the SQL path hits an unexpected blocker (Turso shell auth issues, ALTER TABLE quirks, whatever), the planner can fall back to manually editing each of the ~20 rows through the UI — but this requires temporarily exposing the `isImported` field in the edit form. Prefer the SQL path; treat UI fallback as contingency.
- **Player count filter literal is 2/3/4** — User's original argument said "2 or 4" but agreed 3 should be included for completeness. No 1-player filter.
- **"Players" filter = participated in the game**, not just winners. User's exact phrasing: "players (which players played in the game but did not necessarily win)". This is why the filter source is ALL participants, not just winners.

</specifics>

- **Filter by `wonByCombo = true`** — Not requested by the user, but a natural future filter. Not in 6.1 scope.
- **Filter by date range** — Natural extension of the filter toolbar but not in 6.1 scope. Phase 7 weekly-frequency stats may drive this later.
- **Hide the filter toolbar when there are zero games** — Nice polish; planner can decide whether to include it or leave the toolbar visible with disabled/greyed controls.
- **Separate Phase 7 "stats filter" integration** — Phase 7 will need its own filter semantics for stats (e.g., "show stats for 4-player games only"). That's NOT 6.1 scope — Phase 7 decides whether to reuse the 6.1 filter state or maintain its own.
- **Admin-only editability of imported games** — Not requested. Current model: any logged-in user can edit any game. If this ever changes (admin-only edit of imported rows to prevent tampering), a later phase can add a middleware check or role field.
- **Tag system for game source** — Rejected in favor of a boolean column. Only relevant if multiple import sources (CSV, API, manual bulk-entry) ever appear.
- **Removing the "Ported from Spreadsheet" note from those 20 rows** — Rejected. User prefers the redundancy; two indicators for one fact is fine.
- **Badge / pill UI indicator for imported games** — Rejected. The flag is intentionally hidden from the UI; the notes text is the only visible signal.
- **Live-only / imported-only filter chip** — Rejected. Filters operate on all games uniformly.
